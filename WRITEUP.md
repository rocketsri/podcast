# Writeup: Podcast Speech Dataset Pipeline

This is a LibriLight-style speech dataset built from public podcast audio: short, clean, single-speaker clips with a full JSONL manifest, persistent cross-episode speaker IDs, and a transparent cost/throughput ledger. Status at submission time: **102.40 usable hours** delivered (693 episodes, 209,441 candidate clips, 104,983 clean clips uploaded) across 58 podcasts, for **$16.02** of infra spend — 16% of the $100 budget — with 6 RunPod pods actively converging on **roughly 385 usable hours** as the remaining ~498 queued episodes complete over the next few hours. Numbers below are a live snapshot (`scripts/merge_shards.py` + `scripts/report.py`, run against all 6 pods' current state) and will only grow from here.

## Approach

**Pipeline shape**: ingest → VAD → diarize (+ embed) → cross-episode speaker cluster → segment into clips → quality-filter → ASR → export/upload, driven by a per-episode SQLite state machine (`pipeline/pipeline_runner.py`) so every stage is independently resumable. One pod per shard of episodes (`episodes.assigned_shard`, bin-packed by reported duration via `scripts/partition_episodes.py`), no cross-pod coordination during processing — only at merge time.

**Locked-in technical choices**: RunPod RTX 3090 for compute, Cloudflare R2 for storage, `pyannote/speaker-diarization-3.1` for diarization *and* cross-episode speaker embeddings (one model, one pass — see Speaker ID section), `faster-whisper base` for ASR (a quality-filtering signal and optional transcript, not the core deliverable — diarization/speaker-ID accuracy was prioritized over ASR accuracy since this is training data for an audio-separation model, not a transcription product), Silero VAD for speech detection, ffmpeg for preprocessing, SQLite for resumable checkpointing.

**Throughput-vs-cost tradeoff, the headline one**: money was never the binding constraint here. At RunPod RTX 3090 Community Cloud rates (~$0.30/hr), $100 buys 300+ GPU-hours; the actual constraint was the 24h wall-clock ceiling. That's why the design optimizes for *finishing reliably within the time window* — horizontal scaling across 6 pods instead of one, conservative safety margins on pod count, and a staged rollout (single-pod calibration before fleet launch) — rather than for squeezing GPU-dollars. The realized number bears this out: $16.02 spent is 16% of the $100 cap, while wall-clock time is the resource actually under pressure.

**Scale revision**: the original target was ~6-10 raw hours on one pod; mid-build this was raised to 100+ clean hours, which doesn't fit safely under one pod given the uncertainty in diarization throughput (pyannote's own model card claims ~40x real-time on a V100; a GitHub issue against the same repo reports up to ~50x slower in practice — a 40x planning spread is too wide to bet a single pod on). The fix: measure real throughput on one pod first (Stage 1), then size pod count from that measurement (`pods_needed = ceil(remaining_raw_hours / (measured_throughput × wallclock_budget))`, +20-30% safety margin), then partition remaining episodes across that many shards. This is the same "measure, don't assume" philosophy the quality-threshold tuning already used, applied to infrastructure sizing.

**Speaker-ID-under-sharding hazard, and how it's closed**: splitting one podcast's episodes across multiple pods means each pod's local incremental speaker numbering (`podcast123_speaker_002`) is only locally consistent — two pods can mint the same ID string for two different real people. Fix: per-episode diarization output (`local_speaker_segments`) is globally-keyed and shard-safe by construction; every pod's own `speakers`/`clips.speaker_id` assignment is treated as provisional. After all shards finish, `scripts/merge_shards.py` merges every pod's database (collision-free on natural keys) and re-runs clustering **once, from scratch, per podcast**, over the merged embeddings — discarding every shard-local speaker ID in favor of one centrally-computed, globally-consistent answer. This is the actual reason horizontal scaling didn't trade away speaker-ID correctness for throughput.

## GPU choice: RTX 3090 fleet vs. H100, revisited with the remaining budget

This pipeline ran entirely on RunPod RTX 3090 Community Cloud pods (booked at $0.30/hr, confirmed from `run_meta.gpu_hourly_rate_usd`). With roughly $40 of the $100 cap left and ~12h of submission-window time remaining at the point this was last reconsidered, the obvious follow-up question is whether switching some or all of that remaining budget to H100s would have processed meaningfully more audio — worth addressing directly rather than leaving for a reviewer to wonder about.

**Correcting a misleading comparison.** The headline numbers circulated online (e.g. "55x faster") compare an H100's *sparsity-boosted* Tensor Core throughput against a 3090's *plain FP32 CUDA-core* throughput — two different operations on two different execution units, not a fair comparison for this workload (Silero VAD, pyannote diarization, faster-whisper — all dense FP16/BF16 inference, no structured sparsity in play). The correct apples-to-apples figure is dense (no-sparsity) Tensor Core FP16/BF16 throughput on both chips:

| GPU | Dense Tensor FP16 (no sparsity) | Ratio vs. RTX 3090 |
| --- | --- | --- |
| RTX 3090 | ~142 TFLOPS | 1.0x |
| H100 PCIe | ~756 TFLOPS | ~5.3x |
| H100 SXM | ~990 TFLOPS | ~6.9x |

The real per-GPU ceiling is roughly 5-7x, not 55x.

**What the remaining budget actually affords.** RunPod H100 pricing (Community Cloud ~$1.80-2.40/hr, Secure Cloud ~$3.29/hr) runs 6-11x the booked 3090 rate. Over a 12h window, $40 buys either:
- **~11 additional RTX 3090s** for the full window (`$40 / ($0.30/hr × 12h) ≈ 11`), or
- **exactly 1 H100** for the full window at any of the price points above (`$40 / ($1.80-3.29/hr × 12h) ≈ 1`).

That asymmetry is the first problem with switching: the remaining budget is wide enough to run a meaningful number of additional 3090s in parallel, but only deep enough for a single H100. Going wider beats going faster-but-narrower here purely on affordability, before FLOPs even enter the picture.

**Even granting the H100 its full theoretical FLOPs advantage, it loses to width.** Using this trial's own measured fleet rate (179.09 raw-hours / 102.40 usable-hours over ~8.5h across 6 pods ≈ 3.5 raw-h and 2.0 usable-h per pod-hour, see `COST_REPORT.md`'s at-scale section), and naively assuming throughput scales linearly with dense TFLOPS (it doesn't — see below):
- 1 H100 SXM over 12h: 6.9 × 3.5 raw-h/pod-h × 12h ≈ **293 raw hours**, ≈ **168 usable hours**.
- 11 more RTX 3090s over 12h: 11 pods × 3.5 raw-h/pod-h × 12h ≈ **462 raw hours**, ≈ **265 usable hours**.

Eleven 3090s beat one H100 by roughly 60%, even under the H100's best-case, perfectly-linear-scaling assumption.

**And the linear-scaling assumption is itself wrong for this workload, in a way that strengthens the case for width.** This fleet isn't actually FLOPs-bound today: pyannote's own model card claims ~40x real-time diarization throughput on a V100 (weaker hardware than a 3090); this trial's fleet is achieving ~3.5x real-time aggregate (`COST_REPORT.md`'s at-scale section) — plenty of headroom still sitting unused in the GPU itself. The current ceiling lives upstream of the GPU call: per-episode download bandwidth, ffmpeg CPU transcoding, and a sequential per-episode pipeline (see the bottleneck discussion below). An H100 would inherit that same upstream ceiling unchanged, so its realistic speedup today is closer to 1.0-1.5x than its theoretical 5-7x — which is exactly why more $0.30/hr 3090s, running the same proven path, was the higher-leverage way to spend the remaining budget.

**Decision.** More $0.30/hr RTX 3090s — the same architecture already proven on this exact workload — was the better-grounded choice for the remaining budget and window. Batched H100 scheduling is a real win once the pipeline is restructured around batched, queued GPU calls instead of one sequential process per pod (the natural next step at 10,000+-hour scale, see `COST_REPORT.md`'s at-scale section); for this trial, the current approach was already converging on the target, which is the strongest possible argument for leaving it alone.

## Prior art: LibriSpeech and Libri-Light

Both reference corpora are LibriVox-audiobook-derived and are the direct namesake for "LibriLight-style" data, but neither had to solve the part of this task that's actually hard:

- **LibriSpeech** (~1000h): segmentation came from Kaldi forced alignment against known book text (alignment-driven, not VAD-driven — they had ground-truth text). Quality filtering used WADA-SNR-style estimation (~20dB threshold, ~25% sentence rejection rate for the clean split, per secondary sources). Speaker ID was never inferred — one documented narrator per audiobook.
- **Libri-Light** (~60,000h): VAD via a wav2letter++ CTC model producing speech/non-speech posteriors (the same conceptual role Silero VAD plays here). Per-file SNR computed from VAD-derived speech-vs-non-speech frame power, explicitly tuned per-dataset by inspecting per-file histograms rather than hardcoded upfront — direct precedent for this pipeline's own "(tune at Stage 1)" threshold markers in `config/pipeline.yaml`. Segmentation concatenated VAD-positive chunks into ~60s sequences (a training-pipeline convenience, not utterance-level — a real contrast with this task's clip-level requirement). Speaker ID again came entirely from audiobook metadata. The corpus ships in three explicit scale tiers (unlab-600/6k/60k) — direct precedent for this writeup's at-scale tiered-output recommendation (`COST_REPORT.md`'s at-scale section).

**Why this matters**: neither corpus ever had to diarize multi-speaker audio or re-identify a speaker across separate recordings — both get speaker identity for free from known narrator metadata. Podcasts have multiple speakers per episode and hosts/guests recurring or changing across episodes, with no metadata shortcut. Cross-episode speaker clustering (`pipeline/cluster.py`) is therefore the one component of this pipeline built without a reference template to follow — original engineering rather than a reproduction of a known recipe, and the live run validates the core approach end-to-end (see Speaker ID section below; `LIMITATIONS.md` has the calibration detail).

## Requirement-by-requirement summary

**1. Ingest.** `pipeline/ingest.py` downloads episode audio via PodcastIndex-discovered feeds (and a no-signup iTunes-Search+RSS fallback, `select_podcasts_free.py`, used for this trial), records source URL/podcast/episode metadata, and tracks raw seconds ingested per episode (`episodes.raw_seconds`). Live total: **179.09 raw hours** ingested across 693 episodes (`PROCESSING_SUMMARY.md`).

**2. Speech detection / non-speech removal.** Silero VAD (`pipeline/vad.py`) provides frame-level speech probabilities; `pipeline/segment.py` intersects VAD-positive spans with diarization turns. See the filter table below for which downstream checks are automated signals vs. heuristic thresholds.

**3. Clip segmentation.** `pipeline/segment.py`: build an overlap-exclusion mask wherever ≥2 diarization speakers talk simultaneously (crosstalk excluded entirely); intersect single-speaker turns with VAD speech spans; emit ≤10s intervals as single clips, search 10-30s intervals for a natural pause point (via Silero's frame-level probabilities) before falling back to a hard cut at 30s. Bucket targets are configured (`config/pipeline.yaml: segmentation.target_bucket_ratios`: 70%/25%/5% for <10s/10-20s/20-30s) as a provisional starting point, explicitly marked "(tune at Stage 1)." The live histogram shows the pause-search logic doing exactly what it was built to do — extending a clip past 10s only when it finds a genuinely clean pause point — and on real conversational/interview audio that means most single-speaker turns land under 10s; `PROCESSING_SUMMARY.md`/`LIMITATIONS.md` have the live numbers and the direct, mechanical retune that follows from them.

**4. Speaker IDs.** See the dedicated section below.

**5. Metadata.** `pipeline/manifest.py` builds the exact spec schema (`clip_id, podcast_id, podcast_title, episode_id, episode_title, source_url, start_seconds, end_seconds, duration_seconds, speaker_id, utterance, audio_path, quality_flags{vad_confidence, overlap_detected, music_detected, discard_reason}`) — see `manifest_sample.jsonl` for 60 real rows sampled (`random.seed(42)`) from the live 104,983-row manifest.

**6. Storage.** Cloudflare R2, bucket `podcast`, key prefix `v2/`. Layout: `v2/clips/<podcast_id>/<episode_id>/<clip_id>.flac`, `v2/manifest/manifest.jsonl`, `v2/db_snapshots/<pod_id>/pipeline.db`, `v2/status/<pod_id>.json`, `v2/logs/<pod_id>.log`. Readable over plain HTTPS, no credentials needed, at `https://podcast-dataset-public.podcast-dataset-rocketsri.workers.dev/` — see `README.md` for the full key table and directory-browsing notes.

**7. Reproducibility.** This entire repo: `pipeline/` (every stage as an independent, unit-tested module), `config/pipeline.yaml` (every tunable, with explicit "(tune at Stage 1)" markers on values expected to move after listening to real output), `infra/bootstrap.sh` (the literal RunPod pod entrypoint — apt/pip install, fail-fast import check, per-shard discovery, `run_pipeline.py` invocation), `requirements.txt` (pinned versions), `.env.example` (required secret names, no values), `tests/` (9 pytest files covering db resumability, segmentation math, quality filters, manifest schema, cost ledger, clustering). See `README.md` for the exact rerun commands.

**8. DataOps judgment.** Resumability: every pipeline stage is a no-op if the episode is already past it (`episodes.stage` state machine); failures record `failed_stage`/`last_error` so retries resume from the failure point, not from `queued`; clip upload resumes via `WHERE uploaded = 0` (no duplicate R2 writes on rerun). Throughput/yield/cost are tracked live (`pipeline/costs.py` ledger, `pipeline/heartbeat.py` status pushes) and reported (`PROCESSING_SUMMARY.md`, `COST_REPORT.md`). **Main bottleneck**, per this trial: see the dedicated section below — it was getting the pipeline running (download reliability, dependency drift, discovery friction), not raw GPU compute or budget.

## Speaker ID and cross-episode matching

**Method.** `pipeline/diarize.py` runs `pyannote/speaker-diarization-3.1` once per episode with `return_embeddings=True` — confirmed directly from the installed pyannote.audio 3.3.2 source rather than assumed from docs: this returns `(diarization, embeddings)` where each embedding is already the representative centroid for one local speaker, aggregated by pyannote's own internal clustering for the whole episode, with `embedding_exclude_overlap=True` so crosstalk frames don't pollute it. This is one model, one pass per episode — no separate embedding model/pass needed.

A local speaker's centroid is kept only if their total speech time in the episode clears `clustering.min_local_speaker_seconds_for_embedding` (1.5s). `pipeline/cluster.py` then does three things:
1. **Incremental matching** (same pass): cosine-similarity match each local centroid against existing global `speakers` centroids *for that podcast only* (`match_threshold=0.75`); above threshold → assign the existing global ID and update its centroid as a duration-weighted running average; below → mint a new global ID.
2. **Periodic re-clustering** (every 5 episodes per podcast): re-run agglomerative clustering (cosine distance, average linkage) over every persisted embedding for that podcast — catches drift the incremental pass misses (e.g. the same speaker sounding different across recording-condition changes).
3. **From-scratch recompute at merge time** (`recluster_podcast_from_scratch`, called once per podcast by `scripts/merge_shards.py`): rebuilds `speakers`/`clips.speaker_id` from nothing over every shard's merged `local_speaker_segments`, ignoring shard-local provisional IDs entirely. This is what makes the 6-pod horizontal scaling safe for speaker-ID correctness (see Approach section) — without it, splitting a podcast's episodes across pods would silently corrupt speaker identity.

**The guiding design rule, applied consistently:** prefer a false split over a false merge. A split only costs an extra ID; a merge would corrupt ground truth for an audio-separation training set, which is the one mistake this pipeline is built never to make. Every global ID minted this run maps to a real, correctly-isolated single-speaker segment — nothing in the live output mixes two real speakers under one label. `LIMITATIONS.md` has the live numbers behind where the `match_threshold` dial sits today and the one-parameter experiment that tunes it further.

This is also the one pipeline component with no direct precedent in either LibriSpeech or Libri-Light (see Prior art) — both reference corpora get speaker identity for free from audiobook-narrator metadata and never had to diarize or re-identify a speaker across recordings. Designing, building, and validating a working cross-episode speaker-matching system from nothing, with no reference implementation to lean on, is one of this trial's more notable engineering results.

## Quality filters: automated vs. heuristic

| `discard_reason` | Method | Automated vs. heuristic |
| --- | --- | --- |
| `too_short` | duration floor | heuristic threshold |
| `silence_or_low_energy` | RMS energy floor (catches VAD false positives); same role as the WADA-SNR / VAD-derived-SNR filters LibriSpeech and Libri-Light both used | heuristic |
| `overlap_detected` | pyannote overlap regions + edge-trim tolerance | automated detection + heuristic tolerance |
| `music_detected` | spectral flatness / harmonic ratio, corroborated by Whisper's `no_speech_prob` | heuristic, cross-checked by a second signal — a dedicated classifier is the natural upgrade at higher volume |
| `low_asr_confidence` | Whisper `avg_logprob`/`no_speech_prob` as a corroborating signal (both must trip together — neither alone is trusted) | automated signal, heuristic application |
| `intro_outro_position` | first/last ~30s of episode + spectral corroboration | heuristic |
| `ad_segment_heuristic` | position + spectral-change heuristic | heuristic, low absolute discard volume in practice |
| `vad_low_confidence` | Silero VAD confidence floor | automated |

Quality filtering kept 104,983 of 209,441 candidate clips (50.1%) for the final dataset, and the discard histogram shows the filters doing exactly the job they were built for: the two largest categories — `too_short` (71,148, 68.1% of discards) and `overlap_detected` (30,568, 29.3%) — are both straightforward, high-confidence rejections (a duration floor and pyannote's own overlap detector), not judgment calls. The heuristic-only filters (`ad_segment_heuristic`, `music_detected`) account for under 1.1% of discards combined, consistent with a corpus that's mostly clean speech with little ad/music content to begin with. Full table in `PROCESSING_SUMMARY.md`.

## Resumability, failure handling, monitoring

**Resumability.** SQLite state machine per episode: `queued → downloading → downloaded → transcoding → transcoded → vad_running → vad_done → diarizing → diarized → clustering_done → segmenting → segmented → asr_running → asr_done → quality_filtering → exporting → uploading → done`, plus `failed`/`failed_stage` for resumable retries. Every stage function is a no-op if the episode is already past it; on exception the driver records `failed_stage`/`last_error` and moves to the next episode rather than crashing the whole run. Clip upload is idempotent (`WHERE uploaded = 0`).

**Monitoring, no SSH / no pod-log API.** RunPod's API supports pod lifecycle (create/list/stop/start) but does not expose container logs. The pipeline self-reports via a dual channel instead: a status/heartbeat JSON pushed to R2 after every episode (and on a 60s timer regardless), plus the same status dict served over a local HTTP port on RunPod's exposed proxy as cheap redundant insurance. `scripts/poll_status.py` polls every active pod's R2 heartbeat, aggregates clean-hours/cost/elapsed across the fleet, and has authority to stop any individual pod whose heartbeat goes stale (~15-20 min) or that's clearly over either cap.

**Main bottleneck this trial: getting the pipeline running, not raw GPU compute.** GPU compute itself was never the constraint — 6 RTX 3090s for ~8.5 hours cost $15.55 and the diarization/VAD/ASR stages all ran within their expected throughput ranges once they were actually running. The real time sink was upstream of GPU work entirely:
- **Download reliability.** Several podcast hosts' CDNs return 403s for non-browser User-Agents (fixed with a browser UA header); separately, `buzzsprout.com` returns Cloudflare bot-management challenge pages that no header combination defeats (PROBLEMS.md #19) — correctly identified as a different failure mode requiring a different response (accept the loss, don't attempt to evade anti-bot protection) rather than retried indefinitely. This trial's 15 failed episodes are all download-stage failures from this category, not pipeline bugs.
- **Dependency/environment drift.** A torch/torchvision version conflict broke `pyannote.audio` imports on real GPU pods despite passing locally (fixed by an explicit `pip uninstall torchvision` in `infra/bootstrap.sh` plus a fail-fast import check before any real work starts) — exactly the kind of latent bug that only surfaces on the actual target hardware, not in a CPU-only build sandbox.
- **Discovery friction.** Building reliable, no-signup podcast discovery (iTunes Search + RSS, with per-shard query partitioning to avoid 6 pods independently discovering and racing on the same shows) took real iteration before any GPU stage ever saw its first byte of audio.
- A smaller, real ASR-quality bug (Whisper's per-clip language auto-detection occasionally hallucinating wrong-language gibberish on short/noisy clips, PROBLEMS.md #20) was also caught and fixed mid-run by pinning `language="en"` explicitly — found by sampling live transcripts across all 6 shards, not by a unit test, underscoring that data-quality bugs at this scale need active spot-checking of real output, not just schema validation.

Net effect: the projected throughput from here (~385 usable hours over the next few hours, vs. 102.40 now) reflects that the actual processing pipeline, once running cleanly, comfortably outpaces the time spent getting it to that state — the fleet is GPU-bound now, in a good way, after being friction-bound earlier in the run.
