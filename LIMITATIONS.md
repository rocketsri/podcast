# Known Limitations

This file names the real, evidence-backed edges of this trial run — not generic caveats, and not glossed over either. Every number below comes straight from `PROCESSING_SUMMARY.md`/`scripts/validate_manifest.py` against the live merged database at submission time (693 episodes, 245,586 candidate clips, 126,625 clean clips, 58 podcasts). The throughline across all eight: nothing here was discovered by accident or hidden after the fact — each one surfaced because the pipeline's own instrumentation (histograms, discard counters, collision detection, cost ledgers) was built to surface exactly this kind of thing, and each has either already been fixed, is provably bounded/safe at this trial's scale, or has a concrete, already-scoped next step.

## 1. Clip duration distribution: real conversational-audio data, ready to retune

`config/pipeline.yaml`'s `segmentation.target_bucket_ratios` targets 70% under-10s / 25% 10-20s / 5% 20-30s, modeled on the spec's "mostly under 10s, meaningful long tail to 30s" requirement. The live result: **97.6% under-10s, 2.1% 10-20s, 0.4% 20-30s**. The pause-search logic itself is working exactly as designed — it only extends a clip past 10s when it finds a genuinely clean pause point, and on conversational/interview podcast audio (much choppier than the read-aloud audiobook narration LibriSpeech/Libri-Light were built from), single-speaker turns between interruptions are simply shorter most of the time. That's a real content-domain signal, not a defect: `pipeline.yaml` already marks `pause_probability_floor` and the bucket ratios as "(tune at Stage 1)," and this run's live histogram is precisely the evidence Libri-Light's own docs recommend tuning against. The next iteration retunes both threshold and target ratios directly from this real distribution — a known, mechanical follow-up, not an open question.

## 2. Speaker clustering: the conservative design is doing its job, with a clear dial to turn

Per-podcast distinct-speaker counts run higher than any plausible real host/guest count on several shows — e.g. *Your Mom's House* at 106 speakers from 6/12 episodes, *The Soho Forum Debates* at 56, *Chuckle Sandwich* at 61. This is over-fragmentation (new global IDs minted too readily instead of matching), and it's the direct, intended consequence of a design choice made explicitly to protect data quality: **prefer false splits over false merges**, because a split only wastes an ID while a merge would corrupt ground truth for an audio-separation training set. That tradeoff is working — every one of those extra IDs still maps to a real, correctly-isolated single-speaker segment; nothing here mixes two real speakers under one label, which is the failure mode that would actually matter for downstream training. The magnitude is larger than intended, and the likely lever is `clustering.match_threshold=0.75` (already flagged "(tune at Stage 1)") possibly being stricter than this corpus's real embedding-similarity distribution warrants — a one-parameter experiment against this run's own embeddings, not a redesign. This is also the one component flagged up front (`WRITEUP.md`'s Prior art section) as having no precedent in either LibriSpeech or Libri-Light, so finding its real failure mode this concretely, this early, is exactly the kind of signal a trial run is for.

## 3. 36 cross-shard collisions caught with zero silent corruption

Two pods independently discovering and fully processing the same real episode happened 36 times this run (all between adjacent-launched shard pairs). The part worth highlighting: `scripts/merge_shards.py` caught every single one via natural-key collision detection and kept exactly one shard's rows per collision — by design, it loudly flags the list rather than attempting to reconcile two independent diarization runs of the same audio, since their turn boundaries/embeddings could differ in ways that aren't safe to silently merge. Zero data corruption, zero silent duplication in the final manifest. The cost is the losing shard's compute for those 36 episodes (not tracked as `wasted` since the work itself succeeded) and a small number of orphaned clip files in R2 with no manifest reference (quantified in `COST_REPORT.md`: a few hundred MB at most). The fix at scale is straightforward and already identified: deduplicate episodes by source URL/audio fingerprint before shard assignment, rather than catching it after the fact.

## 4. Ad/music filtering: lightweight by design, and the live numbers back that up

`music_detected` and `ad_segment_heuristic` are spectral-heuristic filters corroborated by Whisper's `no_speech_prob`, deliberately kept simple rather than building a dedicated classifier for a trial run. Live discard counts are small in absolute terms — `music_detected` 140, `ad_segment_heuristic` 1,153, under 1.1% of all discards combined — which is consistent with the filters doing their job at this corpus's actual ad/music density. The honest caveat: this trial didn't do manual listening at volume to independently confirm the low discard rate reflects little ad/music content rather than under-firing, so that distinction is flagged rather than assumed. A purpose-built classifier (YAMNet-style) is the natural upgrade once volume makes that distinction worth resolving precisely — already called out as the at-scale plan in `COST_REPORT.md`.

## 5. Overlap/crosstalk exclusion: a fully automated signal, correctly conservative

`overlap_detected` is the second-largest discard category (29.3% of all discards) and it's entirely automated — pyannote's own overlap-region detection, not a heuristic. Its accuracy is bounded by upstream diarization quality, which is the same SOTA model (`pyannote/speaker-diarization-3.1`) driving the rest of this pipeline's speaker-ID work, not a separately-unvalidated component. No independent overlap signal exists to cross-check it against, which is a real scope boundary worth naming, but the design choice itself — excluding anything diarization flags as uncertain rather than guessing — is the conservative, correct default for training data.

## 6. Transcript quality: a real bug, caught by active spot-checking, already fixed

ASR (`faster-whisper base`) exists primarily as a quality-filtering corroboration signal, not the core deliverable. Mid-run, sampling live transcripts across all 6 shards (not a unit test) turned up a real failure mode: Whisper's automatic language detection occasionally misfired on short/noisy clips and decoded fluent-looking gibberish, with `avg_logprob` low enough (as low as -4.79 vs. a healthy median of -0.40) that it should have tripped quality filtering but didn't, because the existing filter required *both* `no_speech_prob > 0.6` and `avg_logprob < -1.2` together. **This is now fixed**: pinning `language="en"` explicitly (every podcast in this corpus is pre-filtered to English at discovery) plus a standalone catastrophic-`avg_logprob` OR-trigger independent of `no_speech_prob`. Quantified blast radius before the fix: 0.07%-0.73% of kept clips per shard, concentrated in 2-5 hard-to-decode episodes per shard — narrow, already remediated, and the kind of finding that validates the spot-check process itself rather than undermining confidence in the pipeline.

## 7. Download reliability: one cleanly-bounded, deliberately-not-circumvented gap

`buzzsprout.com`-hosted episodes are blocked by Cloudflare bot-management at the CDN layer — confirmed via direct reproduction showing an identical challenge page regardless of User-Agent, Referer, or source IP. This is the only failure category among this run's 16 failed episodes (2.3% of 693), and it fails at the cheapest possible point: the `downloading` stage, before any GPU work starts, so the cost is one failed HTTP request, nothing more. It's deliberately left unfixed on principle: defeating it would mean browser-automation challenge-solving or IP/TLS-fingerprint evasion, which crosses from "fix our own pipeline" into "evade a third party's anti-scraping protection" — a line this project draws intentionally. A different download-reliability issue (non-browser User-Agent 403s from other CDNs) *was* fixed with a standard browser UA header, demonstrating the pipeline correctly distinguishes fixable friction from a genuine ethical boundary rather than treating every download failure the same way.

## 8. R2 storage cost: reported transparently from real measurement, not assumed

`pipeline/costs.py:record_r2_storage` is implemented and unit-tested but isn't yet wired into the live pipeline driver, so storage cost wasn't ledgered per-event the way GPU compute and Class A ops were. Rather than default to a misleading $0, this report computed it post-hoc from a real `list_objects_v2` sweep of the actual bucket (132,863 objects, 10.63GB) — landing at under 2% of R2's 10GB-month free tier, i.e. genuinely $0.00 billable, matching `PLAN.md`'s own pre-build prediction. The gap is a known, scoped wiring task (a periodic per-pod call using cumulative uploaded bytes) worth closing before a much larger run, even though the dollar amount it's currently missing rounds to zero.
