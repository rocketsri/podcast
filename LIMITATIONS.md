# Known Limitations

This file names real, evidence-backed limitations from this trial run, not generic caveats. Where a number is cited, it comes from `PROCESSING_SUMMARY.md`/`scripts/validate_manifest.py` output against the live merged database at submission time (693 episodes, 209,441 candidate clips, 104,983 clean clips, 58 podcasts), not a hypothetical.

## 1. Clip duration distribution skews far shorter than the configured target

`config/pipeline.yaml`'s `segmentation.target_bucket_ratios` targets 70% under-10s / 25% 10-20s / 5% 20-30s — modeled on the spec's "mostly under 10s, meaningful long tail to 30s" requirement. The live result: **97.6% under-10s, 2.0% 10-20s, 0.4% 20-30s** (`PROCESSING_SUMMARY.md`'s duration-distribution table) — the 10-30s long tail the spec explicitly asks for is at 2.4% combined, an order of magnitude under its 30% target.

**Root cause hypothesis**: `segment.py`'s pause-search logic only extends a clip into the 10-30s range when it finds a "clean" pause point (Silero frame-probability below `pause_probability_floor` in the search window) to cut at; on conversational/interview podcast audio — much choppier and more interruption-heavy than read-aloud audiobook narration (the source material both LibriSpeech and Libri-Light were built from) — single-speaker turns between interruptions are simply short most of the time, so there are fewer multi-10s spans available to extend into the long tail in the first place, independent of the pause-detection logic itself. This is a real content-domain mismatch between the bucket-ratio targets (calibrated for audiobook-style continuous narration) and actual podcast conversational cadence, not a bug in the cut-point search. `pipeline.yaml` already marks `pause_probability_floor` and the bucket ratios themselves as "(tune at Stage 1)" — this trial's live numbers are exactly the kind of histogram evidence Libri-Light's own docs recommend tuning against, and the next iteration should retune both threshold and target ratios from this real distribution rather than the pre-Stage-1 assumed values.

## 2. Speaker clustering shows clear signs of over-fragmentation on several podcasts

Per-podcast distinct-speaker counts in `PROCESSING_SUMMARY.md` are implausibly high relative to any plausible real host/guest count for several shows, e.g.:

| Podcast | Episodes done | Distinct speakers |
| --- | --- | --- |
| Your Mom's House with Christina P. and Tom Segura | 6/12 | **106** |
| The Soho Forum Debates | 5/12 | 56 |
| Chuckle Sandwich | 5/12 | 61 |
| Fast Soul Music Podcast | 7/12 | 58 |
| Navigating Narcissism with Dr. Ramani | 3/12 | 18 |

These are near-certainly **over-fragmentation** — the cross-episode matcher minting new global speaker IDs too readily instead of matching a returning host/guest to their existing centroid — not 50-100+ genuinely distinct people appearing across a handful of comedy/interview episodes. The design deliberately biases toward this failure mode over its opposite (see `WRITEUP.md`'s speaker-ID section: "prefer false splits over false merges, since merges corrupt ground truth for an audio-separation training set"), but the live magnitude here is larger than that tradeoff intended to accept. Plausible contributing factors, not yet isolated: `clustering.match_threshold=0.75` may simply be too strict for this corpus's actual embedding-similarity distribution (another "(tune at Stage 1)" value this run's real data should inform); short single-speaker turns (the same choppiness driving limitation #1) may be producing noisier, less representative centroids that fail to match even the same real speaker episode-to-episode; and recording-condition variance (mic, room, episode-to-episode loudness normalization) across a podcast's run could be pushing the same speaker's embedding further from their established centroid than the threshold tolerates. This is exactly the kind of cross-episode speaker re-identification problem flagged in `WRITEUP.md`'s Prior art section as having no precedent in either LibriSpeech or Libri-Light (both get speaker identity for free from audiobook-narrator metadata) — it remains the least-validated, highest-uncertainty component of this pipeline, and this trial's own data shows that uncertainty manifesting concretely rather than just theoretically.

## 3. 36 cross-shard episode collisions are flagged but not auto-resolved

Two different pods can independently discover and fully process the same real episode (different `episode_id` partitioning didn't fully prevent this — discovery itself raced across shards in 36 cases this run, all between adjacent-launched shard pairs: shard-1/shard-3, shard-2/shard-3, shard-3/shard-4). `scripts/merge_shards.py` detects these via natural-key collision during merge (`INSERT OR IGNORE`) and keeps only one shard's episode/clip rows — by design, per the script's own module docstring, it loudly flags the collision list rather than attempting to reconcile `local_speaker_segments` between the two shards' independent diarization runs of the "same" audio. This is a correct, conservative choice (the two shards' diarization output for nominally the same episode could differ in turn boundaries/embeddings in ways that aren't safe to silently merge), but it means: (a) the "losing" shard's compute for those 36 episodes was wasted work, not reflected as `wasted` in the cost ledger since the work itself succeeded, just got discarded at merge time; (b) the corresponding orphaned clip files remain in R2 with no manifest reference (see `COST_REPORT.md`'s note on the 110,989-vs-104,983 clip-object gap). A production version of this pipeline should deduplicate episodes by source URL/audio fingerprint *before* assigning shards, not rely on catching the collision after both shards have already done the work.

## 4. Ad/music/intro-outro filtering is heuristic-only, with no dedicated classifier

Per the filter table in `WRITEUP.md`: `music_detected` and `ad_segment_heuristic` are both explicitly the weakest filters in the stack — spectral-flatness/harmonic-ratio heuristics corroborated by Whisper's `no_speech_prob`, with no purpose-built music or ad classifier anywhere in the pipeline. Live discard counts for these are small in absolute terms (`music_detected` 134, `ad_segment_heuristic` 971, out of 104,458 total discards — under 1.1% combined), which keeps the *known* damage small at this trial's scale, but a low discard rate from a weak filter is also consistent with the filter simply under-firing (ads/music slipping through into kept clips uncaught) rather than there genuinely being little ad/music content in the source audio — this trial cannot distinguish those two explanations without manual listening, which wasn't done at this volume.

## 5. Overlap/crosstalk exclusion depends entirely on diarization accuracy

`overlap_detected` (29.3% of all discards, the second-largest discard category after `too_short`) is automated — driven directly by pyannote's own overlap-region detection — but its accuracy is a strict function of upstream diarization quality. Any diarization error (a turn boundary placed wrong, a speaker mislabeled) propagates silently into either over-discarding clean single-speaker audio as falsely "overlapping," or under-discarding genuine crosstalk that diarization missed. No independent overlap-detection signal exists in this stack to cross-check pyannote's own output against.

## 6. Transcript quality is a secondary signal, not a verified deliverable

ASR (`faster-whisper base`) exists in this pipeline primarily to corroborate quality-filtering decisions (`no_speech_prob`, `avg_logprob`), with the transcript itself included in the manifest because the spec's schema requires an `utterance` field — it was never the core deliverable, and transcript accuracy was not independently measured against ground truth at any point in this trial. One concrete failure mode was caught and fixed mid-run (`PROBLEMS.md` #20): Whisper's per-clip automatic language detection occasionally misfired to a wrong language on short/noisy/ambiguous clips and decoded fluent-looking gibberish in it, with `avg_logprob` low enough (as low as -4.79 vs. a healthy-clip median of about -0.40) that it should have tripped quality filtering but didn't, because the existing `low_asr_confidence` filter required *both* `no_speech_prob > 0.6` and `avg_logprob < -1.2` to fire, and these gibberish clips kept a low `no_speech_prob` (genuine speech was present) even with a catastrophic `avg_logprob`. Fixed by (a) pinning `language="en"` explicitly (every podcast in this corpus is pre-filtered to English at discovery time) and (b) adding a standalone catastrophic-`avg_logprob` OR-trigger independent of `no_speech_prob` (`quality.catastrophic_avg_logprob_floor=-2.0`). Quantified blast radius before the fix: 0.07%-0.73% of kept clips per shard, concentrated in 2-5 specific hard-to-decode episodes per shard — a real but narrow failure mode, not systemic, and already remediated. Disclosed here as evidence of the *kind* of transcript-quality issue this pipeline can hit, not as an open problem.

## 7. Download reliability has one accepted, unfixed gap

`buzzsprout.com`-hosted episodes are blocked by Cloudflare bot-management at the CDN layer (not buzzsprout's own application logic) — confirmed via direct `curl` reproduction showing an identical challenge-page response regardless of User-Agent, Referer, or source IP (`PROBLEMS.md` #19). This is deliberately **not** worked around: defeating it would require browser automation/challenge-solving or IP/TLS-fingerprint evasion, which crosses from "fix our own pipeline" into "evade a third party's anti-scraping protection," and this project draws the line there. The failure is cheap and clean regardless — it happens at the `downloading` stage before any GPU work starts, so blocked episodes cost nothing but a failed HTTP request, and this trial's 15 failed episodes (all at the `downloading` stage) are entirely this category.

## 8. Cost-ledger storage line item was reconstructed post-hoc, not ledgered live

`pipeline/costs.py:record_r2_storage` is implemented and unit-tested but is never actually invoked from the live pipeline driver (`pipeline_runner.py`) — so this run's R2 storage cost was never logged as a `cost_events` row the way GPU compute and Class A ops were. For this submission it was computed after the fact from real measured bucket bytes (a live `list_objects_v2` sweep: 8.87GB across 111,008 objects) rather than assumed zero — see `COST_REPORT.md`'s note — and the result is genuinely negligible (~0.1% of the R2 free tier) at this trial's scale. At a much larger scale this gap should be closed by actually wiring `record_r2_storage` into the pipeline (e.g. a periodic per-pod call using cumulative uploaded bytes), since post-hoc reconstruction doesn't scale as a habit even though the dollar amount it's currently missing does not matter yet.
